H-01 · CSP permite unsafe-inline en script-src — XSS sin proteccion real OWASP A05 ▶
Ubicacion
📄 next.config.ts:32–34
Impacto

Cualquier XSS en la app permite ejecucion arbitraria de JavaScript. La CSP queda inutilizada como defensa secundaria.

Codigo actual (problematico)
// next.config.ts lineas 32-34
process.env.NODE_ENV === 'development'
  ? "script-src 'self' 'unsafe-inline' 'unsafe-eval'"
  : "script-src 'self' 'unsafe-inline'"  // unsafe-inline en PRODUCCION
Correccion recomendada — nonce-based CSP
// 1. En el middleware (proxy.ts) generar nonce por request:
import crypto from 'crypto'
const nonce = crypto.randomBytes(16).toString('base64')
response.headers.set('x-nonce', nonce)

// 2. En next.config.ts usar nonce dinamico:
"script-src 'self' 'nonce-{NONCE}'"

// Next.js 13+ inyecta el nonce automaticamente en Scripts internos
// si se configura correctamente en el middleware.
Alternativa practica a corto plazo

Si implementar nonces es complejo para el sprint actual, documentar explicitamente el riesgo aceptado y priorizar para el siguiente release. El riesgo es bajo si no hay inputs que rendericen HTML crudo.

H-02 · Sin rate limiting en login/signup — fuerza bruta posible OWASP A07 ▶
Ubicacion
📄 app/actions/auth.ts:18 login() 📄 app/actions/auth.ts:36 signup()
Escenario de ataque

Un atacante puede enviar miles de POSTs al server action de login para adivinar contrasenas. Supabase tiene proteccion basica pero configurable. Sin limite en la capa de aplicacion.

Correccion rapida (Supabase nativo)
  • Supabase Dashboard → Auth → Settings → Rate Limits
  • Activar limite de intentos de sign-in y habilitar CAPTCHA (integra hCaptcha nativamente)
Correccion en codigo (Upstash Redis)
import { Ratelimit } from '@upstash/ratelimit'
import { Redis } from '@upstash/redis'

const ratelimit = new Ratelimit({
  redis: Redis.fromEnv(),
  limiter: Ratelimit.slidingWindow(10, '1 m'),
})

export async function login(formData: FormData) {
  const ip = headers().get('x-forwarded-for') ?? 'anon'
  const { success } = await ratelimit.limit(ip)
  if (!success) return { error: 'Demasiados intentos. Espera 1 minuto.' }
  // ... resto del login
}
H-03 · NEXT_PUBLIC_SITE_URL tiene espacio inicial — OAuth redirige a localhost en produccion CWE-547 ▶
Ubicacion
📄 .env.local:5 📄 app/auth/callback/route.ts:4
Impacto

Tras login con magic link o OAuth, el usuario es redirigido a http://localhost:3000/dashboard en lugar del dominio real. La autenticacion queda rota en produccion.

Evidencia
# .env.local linea 5 (nótese el espacio antes del nombre)
 NEXT_PUBLIC_SITE_URL=http://localhost:3000
^-- espacio aqui hace que Node.js lea la clave como " NEXT_PUBLIC_SITE_URL"

# app/auth/callback/route.ts:4
const SITE_URL = process.env.NEXT_PUBLIC_SITE_URL || 'http://localhost:3000'
# -> process.env.NEXT_PUBLIC_SITE_URL === undefined
# -> SITE_URL siempre = 'http://localhost:3000' en cualquier entorno
Correccion — dos pasos
# 1. En .env.local: quitar el espacio inicial
NEXT_PUBLIC_SITE_URL=http://localhost:3000

# 2. En el servidor de produccion (Vercel / Railway / etc.):
NEXT_PUBLIC_SITE_URL=https://reykobarbershop.com

M-02 · getBarberBookedSlots no verifica autenticacion antes de llamar el RPC CWE-284 ▶
Ubicacion
📄 app/actions/appointments.ts:138–150
Contexto

Esta funcion es usada por el calendario de reservas publico. La disponibilidad de franjas es intencionalmente publica. RLS deberia proteger datos sensibles dentro del RPC.

Riesgo concreto

Si el RPC get_barber_booked_slots devuelve IDs de usuarios o informacion identificable ademas de booleanos de ocupacion, cualquier visitante podria extraer datos de clientes. Verificar en Supabase que el RPC solo retorna timestamps sin datos personales.

Accion requerida
-- Verificar en Supabase SQL Editor:
-- El RPC solo debe retornar timestamps de slots ocupados, sin user_id ni otros datos.
-- Ejemplo de implementacion segura:
CREATE OR REPLACE FUNCTION get_barber_booked_slots(p_barber_id uuid, p_slots timestamptz[])
RETURNS timestamptz[] AS $$
  SELECT ARRAY(
    SELECT appointment_date FROM appointments
    WHERE barber_id = p_barber_id
      AND appointment_date = ANY(p_slots)
      AND status NOT IN ('cancelled')
  );
$$ LANGUAGE sql SECURITY DEFINER;

M-07 · Sin politica RLS DELETE en tabla notifications — datos sin poder limpiar CWE-400 ▶
Descripcion

La tabla notifications tiene politicas SELECT, INSERT y UPDATE pero no DELETE. Los administradores no pueden eliminar notificaciones via el cliente anon. Con el tiempo se acumularán miles de registros afectando el rendimiento.

Correccion — ejecutar en Supabase SQL Editor
CREATE POLICY "notif_delete" ON public.notifications FOR DELETE TO authenticated
  USING (public.is_admin());
